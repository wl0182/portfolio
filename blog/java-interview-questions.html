Java Interview Questions

50 concise Q&A to refresh core Java knowledge before your interview.

← Back to Blog

  1. 1) JDK vs JRE vs JVM?

    The JVM (Java Virtual Machine) executes Java bytecode (.class). The JRE (Java Runtime Environment) = JVM + core libraries (rt.jar, modules) and resources required to run Java apps. The JDK (Java Development Kit) = JRE + developer tools like javac, javadoc, jar, and monitoring tools. You develop with the JDK and deploy with a JRE-equivalent (in modern Java, the modular runtime image).

    # compile and run
javac Hello.java
java Hello

  2. 2) What are the four OOP pillars?

    Encapsulation hides implementation via access modifiers and exposes behavior; Inheritance reuses and specializes behavior; Polymorphism lets a single interface represent many concrete types; Abstraction focuses on essential behavior and hides details, often via interfaces/abstract classes.

    interface Shape { double area(); }
class Circle implements Shape { double r; Circle(double r){this.r=r;} public double area(){return Math.PI*r*r;} }

  3. 3) Why is String immutable?

    Immutability ensures security (class loaders/URLs can’t be altered), enables string interning (pool caching), thread-safety without locks, and stable hash codes for map keys. Operations like concatenation create new instances.

    String a = "hi"; String b = a.replace("h", "H"); // a remains "hi"

  4. 4) StringBuilder vs StringBuffer?

    Both are mutable string builders. StringBuffer is synchronized (thread-safe, slower). StringBuilder is not synchronized (faster) and is preferred in single-threaded contexts.

    StringBuilder sb = new StringBuilder().append("Hello ").append("World");

  5. 5) equals() and hashCode() contract?

    If a.equals(b) is true then a.hashCode()==b.hashCode() must be true. Equal objects must have equal hash codes. Violating this breaks hash-based collections. Always override both together for value objects.

    record Point(int x,int y){}

  6. 6) List vs Set vs Map?

    List keeps order and allows duplicates; Set enforces uniqueness; Map stores key→value pairs with unique keys. Choose based on whether you need ordering, uniqueness, or key-based access.

  7. 7) ArrayList vs LinkedList?

    ArrayList offers O(1) random access, amortized O(1) append, but O(n) middle inserts/removals. LinkedList offers O(1) add/remove at ends, but O(n) access and extra memory for pointers. Prefer ArrayList for most cases.

  8. 8) How does HashMap work?

    It computes a hash from the key, finds the bucket index, and resolves collisions with a linked list or tree bins (since Java 8). It uses equals() to confirm key equality. Good hashCode distribution reduces collisions.

  9. 9) ConcurrentHashMap vs HashMap?

    ConcurrentHashMap supports concurrent reads/writes with internal partitioning and CAS; no null keys/values. HashMap is not thread-safe; concurrent access can corrupt it.

  10. 10) Fail-fast vs fail-safe iterators?

    Fail-fast iterators (e.g., ArrayList) detect structural modifications and throw ConcurrentModificationException. Fail-safe iterators (e.g., CopyOnWriteArrayList) iterate over a snapshot and allow concurrent modifications.

  11. 11) final, finally, finalize?

    final applies to variables (no reassign), methods (no override), classes (no extend). finally is a try-block clause that always runs. finalize() is deprecated; use try-with-resources/cleaners instead.

  12. 12) Checked vs unchecked exceptions?

    Checked exceptions (e.g., IOException) must be declared/handled; represent recoverable conditions. Unchecked (subclasses of RuntimeException) represent programming errors (NPE, IAE) and don’t require declaration.

  13. 13) try-with-resources?

    Automatically closes resources implementing AutoCloseable, even on exceptions, reducing leaks and boilerplate.

    try (var in = Files.newBufferedReader(path)) { return in.readLine(); }

  14. 14) Functional interface?

    An interface with exactly one abstract method (SAM). Enables lambda expressions and method references. Examples: Runnable, Callable, Comparator, and the java.util.function types.

  15. 15) Lambda vs anonymous class?

    Lambdas are concise and target functional interfaces; they capture effectively final variables. Anonymous classes can override multiple methods (when implementing/ extending types) and have their own scope, but are more verbose.

  16. 16) Stream intermediate vs terminal ops?

    Intermediate ops (lazy) return a Stream (map, filter, peek). Terminal ops produce a result/side effect (collect, reduce, forEach) and trigger the pipeline.

    List<Integer> evens = nums.stream().filter(n -> n%2==0).toList();

  17. 17) When to use Optional?

    Use as a return type when a value may be absent; avoid as fields, method params, or collection elements. Prefer Optional.orElseThrow/ifPresent over isPresent/get.

  18. 18) record classes?

    Records are immutable data carriers with auto-generated equals/hashCode/toString and canonical constructors. Use them for DTOs and value types. You can still define custom methods and compact constructors.

  19. 19) var and type inference?

    var infers local variable types from the initializer, improving readability when the type is obvious. It’s not dynamic typing—the inferred type is fixed at compile time.

  20. 20) Switch expressions (Java 14+)?

    Switch expressions use arrow labels and yield values; they avoid fall‑through by default and can be used inline.

    int score = switch(grade){ case A->5; case B->4; default->0; };

  21. 21) Sealed classes (Java 17)?

    Sealed classes/interfaces restrict which classes can extend/implement them (via permits), enabling exhaustive handling and tighter models. Good for algebraic data types.

  22. 22) ClassLoader basics?

    Bootstrap → Platform → Application loaders follow parent delegation to enhance security and consistency. Custom loaders can isolate plugins or load from non-standard sources.

  23. 23) JVM memory areas?

    Heap (objects), Stack (frames per thread), Metaspace (class metadata), Code cache (JIT), and Native memory. GC manages the heap; stack is managed per-thread.

  24. 24) Strong vs soft vs weak vs phantom refs?

    Strong: default, prevents GC. Soft: retained until memory pressure (caches). Weak: collected eagerly (e.g., WeakHashMap keys). Phantom: used with ReferenceQueues for post-mortem cleanup tracking.

  25. 25) What is garbage collection?

    Automatic reclamation of unreachable objects. Modern collectors (G1, ZGC, Shenandoah) aim for low pause times using generational spaces, region-based heaps, and concurrent phases.

  26. 26) volatile keyword?

    volatile enforces visibility and ordering guarantees for a variable across threads. It doesn’t make compound actions atomic; use locks/atomics for that.

  27. 27) synchronized vs ReentrantLock?

    synchronized is simpler (intrinsic monitor), automatically released. ReentrantLock provides features like tryLock, timed lock, fairness, and multiple conditions; but requires manual unlock in finally.

  28. 28) wait/notify/notifyAll usage?

    Call on an object’s monitor inside synchronized blocks. wait() releases the lock and suspends the thread; notify() wakes one waiter; notifyAll() wakes all. Prefer higher-level concurrency utilities when possible.

  29. 29) Thread states?

    NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, TERMINATED. They reflect lifecycle from creation to completion including lock/contention and timed waits.

  30. 30) Executor framework benefits?

    Separates task submission from execution; thread pools improve throughput and resource control. Use ExecutorService for lifecycle, ScheduledExecutorService for periodic tasks.

  31. 31) Future vs CompletableFuture?

    CompletableFuture supports async composition, callbacks, and chaining (thenApply, thenCombine, allOf). Future is a simple handle requiring blocking get() and lacks composition.

  32. 32) What is a deadlock?

    A circular wait where threads hold locks and wait for each other. Avoid with consistent lock ordering, timeouts (tryLock), or reducing lock granularity.

  33. 33) Livelock vs starvation?

    Livelock: threads respond to each other but make no progress (they keep yielding). Starvation: a thread never gets CPU/resources due to unfair scheduling or lock acquisition.

  34. 34) Comparable vs Comparator?

    Comparable defines natural ordering inside the class (compareTo), used by Collections.sort. Comparator defines external ordering; you can chain with thenComparing.

  35. 35) Serialization and transient?

    Serialization converts objects to byte streams. Mark fields transient to skip them (e.g., passwords, derived values). Prefer explicit DTOs or JSON libraries over native serialization for services.

  36. 36) Reflection basics?

    Reflection inspects/modifies classes at runtime (Class, Field, Method). Powerful for frameworks and tooling but can break encapsulation and hurt performance; use judiciously.

  37. 37) What are annotations?

    Metadata attached to code, processed at compile time or runtime (via reflection). Examples: @Override, @Deprecated; custom annotations can drive frameworks and AOP.

  38. 38) Generics and type erasure?

    Generics ensure compile-time type safety. Due to type erasure, type parameters aren’t available at runtime, leading to limitations (no new T()). Use Class<T> tokens when needed.

  39. 39) PECS rule for wildcards?

    Producer Extends, Consumer Super: use ? extends T when you only read (the producer), and ? super T when you insert (the consumer). Helps keep APIs flexible.

  40. 40) Autoboxing pitfalls?

    Auto conversion between primitives and wrappers can cause NPEs (unboxing null), performance overhead, and confusing equality (== compares identity for wrappers).

  41. 41) Why BigDecimal for money?

    Floating point (double/float) is binary-based and can’t exactly represent decimals; BigDecimal supports precise decimal arithmetic. Specify scale/rounding explicitly.

    BigDecimal price = new BigDecimal("19.99").setScale(2, RoundingMode.HALF_UP);

  42. 42) java.time vs java.util.Date?

    java.time API (JSR-310) is immutable, thread-safe, and expressive (Instant, LocalDate, ZonedDateTime). Date/Calendar are mutable and error-prone.

  43. 43) Locale and timezone concerns?

    Always specify ZoneId/Locale when parsing/formatting. Store instants in UTC and render in the user’s timezone. Be careful with DST transitions.

  44. 44) Defensive copying?

    Return copies of mutable fields and copy inputs in constructors to protect invariants and avoid representation exposure.

  45. 45) Immutable vs unmodifiable collections?

    Immutable collections can’t change state; unmodifiable wrappers prevent modification via that reference but the underlying collection might still mutate. Prefer true immutability for safety.

  46. 46) When to use EnumMap/EnumSet?

    When keys/elements are enums. EnumMap uses arrays internally (fast, memory efficient); EnumSet uses bit vectors.

  47. 47) What is a memory leak in Java?

    Unintentional object retention prevents GC and increases heap usage (e.g., static collections, caches without eviction, listeners not removed). Detect with profilers and fix by clearing references or using weak refs.

  48. 48) equals() vs ==?

    == compares references for objects (and primitive values for primitives). equals() compares logical equality when overridden. For strings, use equals, not ==.

  49. 49) Static vs instance members?

    Static members belong to the class (one per class loader); instance members belong to each object. Static methods can’t access instance fields directly because they don’t have a this reference.

  50. 50) Best practices for equals/hashCode/toString?

    Include significant fields, maintain consistency with immutability and the general contracts, and avoid including volatile/derived fields. Consider using records or IDE generation to reduce errors.

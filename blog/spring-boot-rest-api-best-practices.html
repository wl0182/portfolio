Spring Boot REST API Best Practices

Design robust, evolvable APIs with validation, error handling, and versioning.

‚Üê Back to Blog

1) Resource modeling and URLs

Use nouns and hierarchies. Prefer plural collection names and predictable paths.

// Good
GET /api/v1/orders
GET /api/v1/orders/{id}
GET /api/v1/orders/{id}/items

// Avoid verbs and RPC-ish names
POST /api/v1/createOrder  // avoid

2) DTOs and validation

Decouple persistence entities from API contracts. Validate inputs with Bean Validation.

public record CreateOrderItemDTO(
  @NotNull Long menuItemId,
  @Positive int quantity
) {}

public record CreateOrderDTO(
  @NotEmpty List<@Valid CreateOrderItemDTO> items
) {}

@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {
  private final OrderService service;
  public OrderController(OrderService service) { this.service = service; }

  @PostMapping
  public ResponseEntity<OrderResponseDTO> create(@Valid @RequestBody CreateOrderDTO in) {
    OrderResponseDTO out = service.create(in);
    return ResponseEntity.status(HttpStatus.CREATED).body(out);
  }
}

3) Consistent error handling (Problem Details)

Return structured errors. Consider RFC 7807 application/problem+json.

@ControllerAdvice
public class ApiExceptionHandler {
  @ExceptionHandler(MethodArgumentNotValidException.class)
  ResponseEntity<Map<String,Object>> handleValidation(MethodArgumentNotValidException ex) {
    var errors = ex.getBindingResult().getFieldErrors().stream()
        .collect(Collectors.groupingBy(FieldError::getField,
            Collectors.mapping(DefaultMessageSourceResolvable::getDefaultMessage, Collectors.toList())));
    Map<String,Object> body = Map.of(
        "type", "https://example.com/problems/validation",
        "title", "Validation Failed",
        "status", 400,
        "errors", errors,
        "timestamp", Instant.now().toString()
    );
    return ResponseEntity.status(HttpStatus.BAD_REQUEST)
        .contentType(MediaType.APPLICATION_PROBLEM_JSON)
        .body(body);
  }
}

4) Versioning strategy

Prefer URI versioning for simplicity; header versioning for strict contract control.

// URI versioning
GET /api/v1/orders
GET /api/v2/orders  // breaking changes live here

// Header versioning example (custom)
GET /api/orders
Accept: application/vnd.example.order+json;version=2

5) Pagination, filtering, sorting

Expose consistent query parameters and return metadata links when useful.

// Requests
GET /api/v1/orders?page=0&size=20&sort=createdAt,desc&status=READY

// Response shape (example)
{
  "content": [ /* items */ ],
  "page": 0,
  "size": 20,
  "totalElements": 128,
  "totalPages": 7,
  "links": {"next": "/api/v1/orders?page=1&size=20"}
}

6) Security and auth

Use HTTPS, JWT/OAuth2, and scope/role checks in controllers and services.

@RestController
@RequestMapping("/api/v1/admin")
@PreAuthorize("hasRole('ADMIN')")
class AdminController {
  @GetMapping("/stats")
  public StatsDTO stats() { /* ... */ return new StatsDTO(); }
}

// SecurityConfig snippet (Spring Security 6)
@Bean SecurityFilterChain http(HttpSecurity http) throws Exception {
  return http.csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(reg -> reg
                .requestMatchers("/api/v1/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())
            .oauth2ResourceServer(oauth -> oauth.jwt())
            .build();
}

7) Idempotency and safety

GET must be safe; POST should be idempotent when creating with client-provided keys; use Idempotency-Key

// Client sends a unique key per logical create attempt
POST /api/v1/payments
Idempotency-Key: 3a7b...  // repeat with same key returns the same result

8) Documentation (OpenAPI/Swagger)

Generate machine-readable contracts and interactive docs for consumers.

// build.gradle snippet
implementation "org.springdoc:springdoc-openapi-starter-webmvc-ui:2.5.0"
// UI at /swagger-ui.html or /swagger-ui/index.html

9) Stability and backward compatibility

Add fields in a backward-compatible way. Don‚Äôt remove or repurpose fields in-place.

  • Use additive changes in the same version; deprecate before removal in the next major.
  • Document deprecations and timelines in OpenAPI and release notes.

10) Testing and performance

Mix unit, slice (@WebMvcTest), and integration tests; measure latencies.

@WebMvcTest(OrderController.class)
class OrderControllerTest {
  @Autowired MockMvc mvc;
  @MockBean OrderService service;

  @Test void createsOrder() throws Exception {
    when(service.create(any())).thenReturn(new OrderResponseDTO(1L, BigDecimal.TEN));
    mvc.perform(post("/api/v1/orders")
         .contentType(MediaType.APPLICATION_JSON)
         .content("{\"items\":[{\"menuItemId\":1,\"quantity\":2}]}"))
       .andExpect(status().isCreated());
  }
}

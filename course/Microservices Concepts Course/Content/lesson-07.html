Messaging with Apache Kafka

Master asynchronous messaging patterns with Apache Kafka for scalable, decoupled microservices communication.

Introduction

Overview of messaging in microservices

Messaging is a communication pattern where services send messages through intermediary message brokers instead of making direct API calls, enabling asynchronous, decoupled communication between distributed components. In microservices architectures, messaging systems act as the nervous system that connects various services, allowing them to communicate without needing to know each other's exact locations, availability status, or processing capabilities. Messages can carry commands that instruct services to perform actions, events that notify about things that have happened, or data that needs to be shared across service boundaries. This approach creates a more resilient and flexible system architecture compared to synchronous service-to-service communication.

Why async messaging is important for decoupling and scalability

Asynchronous messaging eliminates tight coupling between services by allowing producers to send messages without waiting for consumers to process them, meaning services can operate independently and at their own pace without blocking each other. This decoupling enables better scalability because slow or temporarily unavailable consumers don't affect the performance of message producers, and you can scale producer and consumer services independently based on their specific load patterns. Messaging systems provide built-in buffering capabilities that handle traffic spikes gracefully, allowing producers to continue sending messages even when consumers are overwhelmed, and consumers can process messages when resources become available. Additionally, messaging enables fault tolerance through features like message persistence, retries, and dead letter queues that ensure important messages aren't lost even when individual services fail.

Kafka Basics

Topics: what they are and why they matter

Kafka topics are named channels or categories where messages are published and consumed, serving as the primary organizational unit for message streams in Kafka clusters. Each topic represents a particular type of data or event stream, such as "user-registrations," "order-events," or "payment-notifications," allowing you to logically separate different kinds of messages while maintaining high throughput and parallel processing capabilities. Topics are partitioned across multiple Kafka brokers for scalability and fault tolerance, with each partition maintaining an ordered sequence of messages that can be consumed independently by different consumer instances. The topic-based organization enables multiple producers to write to the same topic simultaneously and multiple consumer groups to read from the same topic with different processing logic, creating flexible pub-sub messaging patterns.

Producer implementation example

Kafka producers are responsible for publishing messages to specific topics, and Spring Boot provides excellent integration through the KafkaTemplate class that simplifies message production with features like automatic serialization, error handling, and delivery confirmations. The producer configuration includes settings for Kafka broker connections, serialization formats for keys and values, and performance tuning parameters like batch size and compression. Producers can send messages synchronously for guaranteed delivery confirmation or asynchronously for higher throughput, and they automatically handle partitioning, retries, and connection management to ensure reliable message delivery to Kafka brokers.

Consumer implementation example

Kafka consumers subscribe to topics and process messages in real-time, with Spring Boot's @KafkaListener annotation providing a declarative way to create message consumers that automatically handle deserialization, offset management, and error handling. Consumer groups allow multiple consumer instances to share the workload of processing messages from a topic, with Kafka automatically distributing partitions among available consumers for parallel processing and load balancing. Consumers can be configured for different processing guarantees, from at-least-once delivery with automatic offset commits to exactly-once processing with manual offset management, depending on your application's consistency requirements.

Producer Service Implementation

Create a robust producer service with comprehensive error handling, metrics, and callback mechanisms for reliable message publishing.

@Service
@Slf4j
public class OrderEventProducer {

    private final KafkaTemplate<String, Object> kafkaTemplate;
    private final MeterRegistry meterRegistry;
    private final ObjectMapper objectMapper;

    public OrderEventProducer(KafkaTemplate<String, Object> kafkaTemplate,
                             MeterRegistry meterRegistry,
                             ObjectMapper objectMapper) {
        this.kafkaTemplate = kafkaTemplate;
        this.meterRegistry = meterRegistry;
        this.objectMapper = objectMapper;
    }

    public CompletableFuture<SendResult<String, Object>> publishOrderCreated(OrderEvent orderEvent) {
        return publishEvent("order-events", orderEvent.getOrderId(), orderEvent, "ORDER_CREATED");
    }

    public CompletableFuture<SendResult<String, Object>> publishOrderStatusChanged(String orderId,
                                                                                  OrderStatus newStatus,
                                                                                  Map<String, Object> metadata) {
        OrderEvent event = OrderEvent.builder()
                .orderId(orderId)
                .eventType("ORDER_STATUS_CHANGED")
                .status(newStatus)
                .metadata(metadata)
                .timestamp(Instant.now())
                .build();

        return publishEvent("order-events", orderId, event, "ORDER_STATUS_CHANGED");
    }

    public CompletableFuture<SendResult<String, Object>> publishOrderCancelled(String orderId, String reason) {
        Map<String, Object> metadata = Map.of(
                "cancellationReason", reason,
                "cancelledAt", Instant.now().toString()
        );

        OrderEvent event = OrderEvent.builder()
                .orderId(orderId)
                .eventType("ORDER_CANCELLED")
                .status(OrderStatus.CANCELLED)
                .metadata(metadata)
                .timestamp(Instant.now())
                .build();

        return publishEvent("order-events", orderId, event, "ORDER_CANCELLED");
    }

    private CompletableFuture<SendResult<String, Object>> publishEvent(String topic,
                                                                        String key,
                                                                        Object event,
                                                                        String eventType) {
        Timer.Sample sample = Timer.start(meterRegistry);

        try {
            // Add correlation ID and tracking headers
            ProducerRecord<String, Object> record = new ProducerRecord<>(topic, key, event);
            record.headers().add("eventType", eventType.getBytes());
            record.headers().add("correlationId", UUID.randomUUID().toString().getBytes());
            record.headers().add("source", "order-service".getBytes());
            record.headers().add("timestamp", Instant.now().toString().getBytes());

            log.info("Publishing {} event for key: {} to topic: {}", eventType, key, topic);

            return kafkaTemplate.send(record)
                    .completable()
                    .whenComplete((result, throwable) -> {
                        sample.stop(Timer.builder("kafka.producer.duration")
                                .tag("topic", topic)
                                .tag("event.type", eventType)
                                .tag("status", throwable == null ? "success" : "error")
                                .register(meterRegistry));

                        if (throwable == null) {
                            log.info("Successfully published {} event for key: {} to topic: {} at offset: {}",
                                    eventType, key, topic, result.getRecordMetadata().offset());

                            meterRegistry.counter("kafka.producer.messages.sent",
                                    "topic", topic, "event.type", eventType).increment();
                        } else {
                            log.error("Failed to publish {} event for key: {} to topic: {}",
                                    eventType, key, topic, throwable);

                            meterRegistry.counter("kafka.producer.messages.failed",
                                    "topic", topic, "event.type", eventType).increment();
                        }
                    });

        } catch (Exception e) {
            sample.stop(Timer.builder("kafka.producer.duration")
                    .tag("topic", topic)
                    .tag("event.type", eventType)
                    .tag("status", "error")
                    .register(meterRegistry));

            log.error("Exception while publishing {} event for key: {} to topic: {}", eventType, key, topic, e);
            meterRegistry.counter("kafka.producer.messages.failed",
                    "topic", topic, "event.type", eventType).increment();

            return CompletableFuture.failedFuture(e);
        }
    }

    @EventListener
    public void handleOrderCreatedDomainEvent(OrderCreatedDomainEvent domainEvent) {
        OrderEvent kafkaEvent = OrderEvent.builder()
                .orderId(domainEvent.getOrderId())
                .userId(domainEvent.getUserId())
                .eventType("ORDER_CREATED")
                .items(domainEvent.getItems())
                .totalAmount(domainEvent.getTotalAmount())
                .status(OrderStatus.PENDING)
                .timestamp(domainEvent.getOccurredAt())
                .build();

        publishOrderCreated(kafkaEvent)
                .exceptionally(throwable -> {
                    log.error("Failed to publish order created event for order: {}",
                             domainEvent.getOrderId(), throwable);
                    // Could trigger compensation or retry mechanisms here
                    return null;
                });
    }

Transactional Producer

Implement transactional messaging to ensure exactly-once delivery semantics and maintain data consistency across database and Kafka operations.

@Service
@Transactional
@Slf4j
public class TransactionalOrderEventProducer {

    private final KafkaTransactionManager transactionManager;
    private final KafkaTemplate<String, Object> kafkaTemplate;
    private final OrderRepository orderRepository;

    public TransactionalOrderEventProducer(KafkaTransactionManager transactionManager,
                                          KafkaTemplate<String, Object> kafkaTemplate,
                                          OrderRepository orderRepository) {
        this.transactionManager = transactionManager;
        this.kafkaTemplate = kafkaTemplate;
        this.orderRepository = orderRepository;
    }

    @KafkaTransactional
    public void createOrderWithEvent(CreateOrderRequest request) {
        try {
            // Database operation
            Order order = Order.builder()
                    .userId(request.getUserId())
                    .items(request.getItems())
                    .totalAmount(calculateTotal(request.getItems()))
                    .status(OrderStatus.PENDING)
                    .createdAt(Instant.now())
                    .build();

            Order savedOrder = orderRepository.save(order);
            log.info("Order saved to database: {}", savedOrder.getId());

            // Kafka operation (within same transaction)
            OrderEvent event = OrderEvent.builder()
                    .orderId(savedOrder.getId())
                    .userId(savedOrder.getUserId())
                    .eventType("ORDER_CREATED")
                    .items(savedOrder.getItems())
                    .totalAmount(savedOrder.getTotalAmount())
                    .status(savedOrder.getStatus())
                    .timestamp(Instant.now())
                    .build();

            kafkaTemplate.send("order-events", savedOrder.getId(), event)
                    .addCallback(
                            result -> log.info("Order event sent in transaction: {}", savedOrder.getId()),
                            failure -> {
                                log.error("Failed to send order event in transaction: {}",
                                         savedOrder.getId(), failure);
                                throw new RuntimeException("Kafka send failed", failure);
                            }
                    );

            log.info("Order creation transaction completed successfully: {}", savedOrder.getId());

        } catch (Exception e) {
            log.error("Order creation transaction failed", e);
            throw new RuntimeException("Transaction failed", e);
        }
    }

    private BigDecimal calculateTotal(List<OrderItem> items) {
        return items.stream()
                .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
                .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

Basic Consumer Service

Implement consumer services with proper error handling, message acknowledgment, and processing logic for different event types.

@Component
@Slf4j
public class OrderEventConsumer {

    private final InventoryService inventoryService;
    private final NotificationService notificationService;
    private final OrderService orderService;
    private final MeterRegistry meterRegistry;

    public OrderEventConsumer(InventoryService inventoryService,
                             NotificationService notificationService,
                             OrderService orderService,
                             MeterRegistry meterRegistry) {
        this.inventoryService = inventoryService;
        this.notificationService = notificationService;
        this.orderService = orderService;
        this.meterRegistry = meterRegistry;
    }

    @KafkaListener(topics = "order-events", groupId = "inventory-service")
    public void handleInventoryReservation(OrderEvent orderEvent,
                                          @Header("kafka_receivedMessageKey") String key,
                                          @Header("kafka_receivedPartition") int partition,
                                          @Header("kafka_offset") long offset,
                                          @Header(value = "eventType", required = false) String eventType,
                                          Acknowledgment acknowledgment) {

        Timer.Sample sample = Timer.start(meterRegistry);

        try {
            log.info("Processing order event: {} from partition: {} offset: {} eventType: {}",
                    orderEvent.getOrderId(), partition, offset, eventType);

            switch (orderEvent.getEventType()) {
                case "ORDER_CREATED":
                    inventoryService.reserveItems(orderEvent.getOrderId(), orderEvent.getItems());
                    log.info("Inventory reserved for order: {}", orderEvent.getOrderId());
                    break;
                case "ORDER_CANCELLED":
                    inventoryService.releaseReservation(orderEvent.getOrderId());
                    log.info("Inventory reservation released for order: {}", orderEvent.getOrderId());
                    break;
                case "ORDER_CONFIRMED":
                    inventoryService.confirmReservation(orderEvent.getOrderId());
                    log.info("Inventory reservation confirmed for order: {}", orderEvent.getOrderId());
                    break;
                default:
                    log.warn("Unknown event type for inventory processing: {}", orderEvent.getEventType());
            }

            acknowledgment.acknowledge();
            meterRegistry.counter("kafka.consumer.messages.processed",
                    "topic", "order-events",
                    "group", "inventory-service",
                    "event.type", orderEvent.getEventType(),
                    "status", "success").increment();

        } catch (InventoryException e) {
            log.error("Inventory processing failed for order: {}, will retry",
                     orderEvent.getOrderId(), e);
            meterRegistry.counter("kafka.consumer.messages.processed",
                    "topic", "order-events",
                    "group", "inventory-service",
                    "event.type", orderEvent.getEventType(),
                    "status", "retry").increment();
            throw e; // Will trigger retry mechanism

        } catch (Exception e) {
            log.error("Unexpected error processing order event: {}", orderEvent.getOrderId(), e);
            acknowledgment.acknowledge(); // Acknowledge to prevent infinite retries
            meterRegistry.counter("kafka.consumer.messages.processed",
                    "topic", "order-events",
                    "group", "inventory-service",
                    "event.type", orderEvent.getEventType(),
                    "status", "error").increment();
        } finally {
            sample.stop(Timer.builder("kafka.consumer.duration")
                    .tag("topic", "order-events")
                    .tag("group", "inventory-service")
                    .tag("event.type", orderEvent.getEventType())
                    .register(meterRegistry));
        }
    }

    @KafkaListener(topics = "order-events", groupId = "notification-service")
    public void handleNotifications(OrderEvent orderEvent, Acknowledgment acknowledgment) {
        try {
            log.info("Processing notification for order event: {} type: {}",
                    orderEvent.getOrderId(), orderEvent.getEventType());

            switch (orderEvent.getEventType()) {
                case "ORDER_CREATED":
                    notificationService.sendOrderConfirmation(orderEvent);
                    break;
                case "ORDER_SHIPPED":
                    notificationService.sendShippingNotification(orderEvent);
                    break;
                case "ORDER_DELIVERED":
                    notificationService.sendDeliveryConfirmation(orderEvent);
                    break;
                case "ORDER_CANCELLED":
                    notificationService.sendCancellationNotification(orderEvent);
                    break;
                default:
                    log.debug("No notification required for event type: {}", orderEvent.getEventType());
            }

            acknowledgment.acknowledge();
            log.debug("Notification processed successfully for order: {}", orderEvent.getOrderId());

        } catch (NotificationException e) {
            log.warn("Notification failed for order: {}, will be retried",
                    orderEvent.getOrderId(), e);
            throw e;
        } catch (Exception e) {
            log.error("Unexpected error in notification processing for order: {}",
                     orderEvent.getOrderId(), e);
            acknowledgment.acknowledge(); // Skip problematic messages
        }
    }

    @KafkaListener(topics = "order-events", groupId = "analytics-service",
                  concurrency = "3",
                  containerFactory = "batchKafkaListenerContainerFactory")
    public void handleAnalytics(List<OrderEvent> orderEvents,
                               List<Acknowledgment> acknowledgments) {
        try {
            log.info("Processing batch of {} order events for analytics", orderEvents.size());

            List<AnalyticsEvent> analyticsEvents = orderEvents.stream()
                    .map(this::convertToAnalyticsEvent)
                    .collect(Collectors.toList());

            analyticsService.processBatch(analyticsEvents);

            // Acknowledge all messages in batch
            acknowledgments.forEach(Acknowledgment::acknowledge);

            log.info("Analytics batch processing completed for {} events", orderEvents.size());

        } catch (Exception e) {
            log.error("Analytics batch processing failed", e);
            throw e; // Will retry entire batch
        }
    }

    private AnalyticsEvent convertToAnalyticsEvent(OrderEvent orderEvent) {
        return AnalyticsEvent.builder()
                .eventId(UUID.randomUUID().toString())
                .orderId(orderEvent.getOrderId())
                .userId(orderEvent.getUserId())
                .eventType(orderEvent.getEventType())
                .totalAmount(orderEvent.getTotalAmount())
                .itemCount(orderEvent.getItems() != null ? orderEvent.getItems().size() : 0)
                .timestamp(orderEvent.getTimestamp())
                .build();
    }
}

Batch Consumer Configuration

Configure batch consumption for high-throughput scenarios where processing multiple messages together is more efficient.

@Configuration
public class BatchConsumerConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, Object> batchKafkaListenerContainerFactory(
            ConsumerFactory<String, Object> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, Object> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);
        factory.setBatchListener(true);
        factory.setConcurrency(3);
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE);
        factory.getContainerProperties().setPollTimeout(3000);

        // Batch configuration
        factory.getContainerProperties().setIdleBetweenPolls(100);
        factory.setCommonErrorHandler(batchErrorHandler());

        return factory;
    }

    @Bean
    public DefaultErrorHandler batchErrorHandler() {
        ExponentialBackOffWithMaxRetries backOff = new ExponentialBackOffWithMaxRetries(2);
        backOff.setInitialInterval(1000L);
        backOff.setMultiplier(2.0);
        backOff.setMaxInterval(5000L);

        return new DefaultErrorHandler(backOff);
    }
}

                                       @Header("kafka_receivedTopic") String topic,
                                       @Header("kafka_receivedPartition") int partition,
                                       @Header("kafka_offset") long offset,
                                       @Header(value = "kafka_dlt-original-topic", required = false) String originalTopic,
                                       @Header(value = "kafka_dlt-exception-message", required = false) String exceptionMessage,
                                       Acknowledgment acknowledgment) {

        try {
            log.error("Processing dead letter message from topic: {} partition: {} offset: {} originalTopic: {} error: {}",
                     topic, partition, offset, originalTopic, exceptionMessage);

            DeadLetterRecord record = DeadLetterRecord.builder()
                    .id(UUID.randomUUID().toString())
                    .originalTopic(originalTopic)
                    .message(message)
                    .errorMessage(exceptionMessage)
                    .receivedAt(Instant.now())
                    .partition(partition)
                    .offset(offset)
                    .status(DeadLetterStatus.RECEIVED)
                    .build();

            messageRecoveryService.saveDeadLetterRecord(record);

            // Send alert for dead letter message
            alertingService.sendDeadLetterAlert(record);

            acknowledgment.acknowledge();
            log.info("Dead letter message processed and saved with ID: {}", record.getId());

        } catch (Exception e) {
            log.error("Failed to process dead letter message from topic: {} partition: {} offset: {}",
                     topic, partition, offset, e);
            acknowledgment.acknowledge(); // Prevent infinite loop
        }
    }

    @KafkaListener(topics = "payment-events.DLT", groupId = "dead-letter-processor")
    public void handlePaymentDeadLetterMessage(@Payload String message,
                                              @Header Map<String, Object> headers,
                                              Acknowledgment acknowledgment) {
        // Similar processing for payment dead letter messages
        handleGenericDeadLetterMessage(message, headers, acknowledgment, "payment-events");
    }

    private void handleGenericDeadLetterMessage(String message,
                                               Map<String, Object> headers,
                                               Acknowledgment acknowledgment,
                                               String expectedOriginalTopic) {
        try {
            log.warn("Processing generic dead letter message for topic: {}", expectedOriginalTopic);

            // Extract common headers
            String originalTopic = (String) headers.get("kafka_dlt-original-topic");
            String exceptionMessage = (String) headers.get("kafka_dlt-exception-message");
            Integer partition = (Integer) headers.get("kafka_receivedPartition");
            Long offset = (Long) headers.get("kafka_offset");

            // Process and store dead letter record
            DeadLetterRecord record = DeadLetterRecord.builder()
                    .id(UUID.randomUUID().toString())
                    .originalTopic(originalTopic)
                    .message(message)
                    .errorMessage(exceptionMessage)
                    .receivedAt(Instant.now())
                    .partition(partition != null ? partition : -1)
                    .offset(offset != null ? offset : -1L)
                    .status(DeadLetterStatus.RECEIVED)
                    .metadata(headers)
                    .build();

            messageRecoveryService.saveDeadLetterRecord(record);
            alertingService.sendDeadLetterAlert(record);

            acknowledgment.acknowledge();

        } catch (Exception e) {
            log.error("Failed to process generic dead letter message", e);
            acknowledgment.acknowledge();
        }
    }
}

Patterns & Best Practices

Pub/Sub pattern and when to use it

The Publish/Subscribe pattern enables one-to-many communication where publishers send messages to topics or exchanges without knowing which subscribers will receive them, creating loose coupling and enabling dynamic system composition. This pattern is ideal for broadcasting events like user registrations, order completions, or system notifications where multiple services need to react differently to the same event without the publisher needing to know about all consumers. Pub/Sub excels in scenarios where you want to add new functionality by simply subscribing to existing events, such as adding analytics, audit logging, or notification services without modifying existing business logic. The pattern also supports fan-out scenarios where a single event triggers multiple workflows, like order processing that simultaneously updates inventory, sends confirmations, and logs analytics data.

Event-driven microservices patterns

Event-driven microservices communicate primarily through domain events that represent meaningful business occurrences, creating architectures where services react to changes rather than being directly invoked by other services. This approach promotes temporal decoupling where services don't need to be available simultaneously, spatial decoupling where services don't need to know each other's locations, and logical decoupling where services react to business events rather than implementation details. Common patterns include event sourcing where state changes are stored as events, CQRS (Command Query Responsibility Segregation) where read and write models are separated, and saga patterns for managing distributed transactions across multiple services. Event-driven architectures enable better scalability, fault tolerance, and system evolution because services can be added, modified, or removed without affecting other services as long as they respect the event contracts.

Tips for reliable messaging (idempotency, retries, DLQs)

Idempotency ensures that processing the same message multiple times produces the same result, which is critical for reliable messaging systems because messages can be delivered more than once due to network issues, retries, or system failures. Implement idempotency by using unique message IDs, database constraints, or application-level deduplication logic that can safely ignore duplicate message processing attempts. Design retry strategies with exponential backoff and maximum retry limits to handle transient failures without overwhelming struggling services, and use circuit breaker patterns to prevent cascading failures when downstream services become unavailable. Dead Letter Queues provide essential safety nets for messages that cannot be processed successfully, enabling manual investigation, system debugging, and message reprocessing after fixing underlying issues, while preventing poison messages from blocking the processing of valid messages in the system.

Order Service - Event Publisher

@Service
@Transactional
@Slf4j
public class OrderService {

    private final OrderRepository orderRepository;
    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public OrderService(OrderRepository orderRepository,
                       KafkaTemplate<String, OrderEvent> kafkaTemplate) {
        this.orderRepository = orderRepository;
        this.kafkaTemplate = kafkaTemplate;
    }

    public Order createOrder(CreateOrderRequest request) {
        // Create order
        Order order = Order.builder()
            .userId(request.getUserId())
            .items(request.getItems())
            .totalAmount(calculateTotal(request.getItems()))
            .status(OrderStatus.PENDING)
            .createdAt(Instant.now())
            .build();

        Order savedOrder = orderRepository.save(order);
        log.info("Order created: {}", savedOrder.getId());

        // Publish event
        OrderEvent event = OrderEvent.builder()
            .orderId(savedOrder.getId())
            .userId(savedOrder.getUserId())
            .eventType("ORDER_CREATED")
            .items(savedOrder.getItems())
            .totalAmount(savedOrder.getTotalAmount())
            .status(savedOrder.getStatus())
            .timestamp(Instant.now())
            .build();

        kafkaTemplate.send("order-events", savedOrder.getId(), event)
            .addCallback(
                result -> log.info("Order event published successfully: {}", savedOrder.getId()),
                failure -> log.error("Failed to publish order event: {}", savedOrder.getId(), failure)
            );

        return savedOrder;
    }

    public Order updateOrderStatus(String orderId, OrderStatus newStatus) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException("Order not found: " + orderId));

        order.setStatus(newStatus);
        order.setUpdatedAt(Instant.now());
        Order updatedOrder = orderRepository.save(order);

        // Publish status change event
        OrderEvent event = OrderEvent.builder()
            .orderId(orderId)
            .userId(order.getUserId())
            .eventType("ORDER_STATUS_CHANGED")
            .status(newStatus)
            .timestamp(Instant.now())
            .build();

        kafkaTemplate.send("order-events", orderId, event);
        return updatedOrder;
    }

    private BigDecimal calculateTotal(List<OrderItem> items) {
        return items.stream()
            .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

Payment Service - Event Consumer & Publisher

@Component
@Slf4j
public class PaymentEventHandler {

    private final PaymentService paymentService;
    private final KafkaTemplate<String, PaymentEvent> kafkaTemplate;

    public PaymentEventHandler(PaymentService paymentService,
                              KafkaTemplate<String, PaymentEvent> kafkaTemplate) {
        this.paymentService = paymentService;
        this.kafkaTemplate = kafkaTemplate;
    }

    @KafkaListener(topics = "order-events", groupId = "payment-service")
    @Retryable(maxAttempts = 3, backoff = @Backoff(delay = 1000, multiplier = 2))
    public void handleOrderCreated(OrderEvent orderEvent) {
        if ("ORDER_CREATED".equals(orderEvent.getEventType())) {
            log.info("Processing payment for order: {}", orderEvent.getOrderId());

            try {
                PaymentResult result = paymentService.processPayment(
                    orderEvent.getOrderId(),
                    orderEvent.getTotalAmount(),
                    orderEvent.getUserId()
                );

                PaymentEvent paymentEvent = PaymentEvent.builder()
                    .paymentId(result.getPaymentId())
                    .orderId(orderEvent.getOrderId())
                    .eventType(result.isSuccessful() ? "PAYMENT_COMPLETED" : "PAYMENT_FAILED")
                    .amount(orderEvent.getTotalAmount())
                    .status(result.getStatus())
                    .timestamp(Instant.now())
                    .build();

                kafkaTemplate.send("payment-events", orderEvent.getOrderId(), paymentEvent)
                    .addCallback(
                        success -> log.info("Payment event published: {}", paymentEvent.getEventType()),
                        failure -> log.error("Failed to publish payment event", failure)
                    );

            } catch (PaymentException e) {
                log.error("Payment processing failed for order: {}", orderEvent.getOrderId(), e);

                // Publish payment failed event
                PaymentEvent failedEvent = PaymentEvent.builder()
                    .orderId(orderEvent.getOrderId())
                    .eventType("PAYMENT_FAILED")
                    .amount(orderEvent.getTotalAmount())
                    .status(PaymentStatus.FAILED)
                    .timestamp(Instant.now())
                    .build();

                kafkaTemplate.send("payment-events", orderEvent.getOrderId(), failedEvent);
                throw e; // Will trigger retry
            }
        }
    }
}

Lesson Summary

In this lesson, we explored comprehensive Apache Kafka implementation for microservices messaging. Here's a complete summary of all concepts and implementation approaches covered:

Messaging Fundamentals

  • Purpose: Asynchronous communication pattern using message brokers for decoupled service interaction
  • Benefits: Eliminates tight coupling, enables better scalability, and provides built-in buffering for traffic spikes
  • Message types: Commands (instruct actions), events (notify occurrences), and data sharing across service boundaries
  • Resilience: Fault tolerance through message persistence, retries, and dead letter topics

Kafka Architecture and Concepts

  • Topics: Named channels for organizing message streams with partitioning for scalability and parallel processing
  • Partitions: Ordered sequences of messages distributed across brokers for high throughput and fault tolerance
  • Producers: Applications that publish messages to topics with automatic serialization and delivery confirmation
  • Consumers: Applications that subscribe to topics and process messages with group-based load distribution

Kafka Configuration

  • Maven dependencies: Spring Kafka starter, actuator, validation, and Jackson for JSON serialization
  • Application properties: Bootstrap servers, serialization settings, and performance tuning parameters
  • Java configuration: Producer factory, consumer factory, and listener container factory with custom settings
  • Topic management: Automatic topic creation with partition count, replication, and retention policies

Advanced Kafka Configuration

  • Performance settings: Batch size, linger time, compression, and memory buffer configuration
  • Reliability settings: Acknowledgment modes, retries, idempotence, and timeout configurations
  • Error handling: Dead letter topic routing, retry policies, and exception classification
  • Monitoring integration: Metrics collection, producer interceptors, and operational visibility

Kafka Producer Implementation

  • Event models: JSON-serializable domain events with proper annotations and versioning
  • Producer service: Robust message publishing with error handling, metrics, and callback mechanisms
  • Message headers: Correlation IDs, event types, source identification, and timestamp tracking
  • Asynchronous publishing: CompletableFuture-based APIs for non-blocking message production

Transactional Messaging

  • Exactly-once semantics: Transactional producers for consistent database and Kafka operations

  • Atomic operations: Combined database writes and message publishing in single transactions

Kafka Consumer Implementation

Advanced Consumer Patterns

  • Manual acknowledgment: Precise control over message acknowledgment timing and conditions

Dead Letter Topic Handling

  • Failed message capture: Automatic routing of permanently failed messages to dead letter topics
  • Recovery mechanisms: Manual investigation, message reprocessing, and system debugging capabilities
  • Alerting integration: Notification systems for operational teams when messages fail
  • Metadata preservation: Original message context, error details, and processing history

Event-Driven Architecture Patterns

Monitoring and Operations

  • Performance optimization: JVM tuning, network configuration, and storage optimization
